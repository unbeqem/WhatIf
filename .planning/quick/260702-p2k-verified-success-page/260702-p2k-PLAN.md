# Quick Task 260702-p2k — Email-verified success page + auto-redirect

## Problem

After clicking the signup confirmation link, `/auth/confirm` verifies the OTP
and redirects straight to `/?confirmed=1` — but the landing never surfaces that
param, so there's zero "you're verified" feedback. Abrupt UX. Failures redirect
to `/login?error=confirmation_failed`, but `/login` doesn't render the error
either, so an expired link fails silently.

## Solution

New client page `app/verified/page.tsx` (brand-matched, Nav/Footer/BackgroundOrbs
like the other auth pages) with two states:

- **Success**: green check, "Email verified", short countdown, auto-redirect via
  `router.replace()` after ~2.8s (with a manual "Continue now" link). Reads a
  sanitized `next` param (default `/decision` — the activation path).
- **Failure/expired**: amber warning, distinct message, CTAs to request a new
  link (`/signup`) / sign in (`/login`). Triggered by `?error=`.

`useSearchParams` is wrapped in a `<Suspense>` boundary (Next 16 requirement).

`app/auth/confirm/route.ts` changes (scoped to the signup/email-confirm path so
password **recovery** — which always carries `next=/reset`, i.e. `target !== "/"`
— is left completely untouched):

- Success default `target === "/"`: redirect to `/verified` instead of `/?confirmed=1`.
- OTP verify failure: non-recovery types redirect to `/verified?error=confirmation_failed`;
  `recovery` keeps the existing `/login?error=confirmation_failed`.
- PKCE (`exchangeCodeForSession`) fallback failure → `/verified?error=confirmation_failed`.

## Verification

- `npx tsc --noEmit` clean
- `npm run build` succeeds; `/verified` builds
- Recovery redirect target (`/reset`) unchanged in the route logic
