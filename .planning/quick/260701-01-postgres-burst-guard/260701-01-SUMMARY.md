# Quick Task 260701-01: Postgres-backed per-IP burst guard — SUMMARY

## What changed
- **`lib/ratelimit.ts`** — `checkBurst` rewritten to count `simulation_usage` rows
  by `ip_hash` in a rolling 60s window via the Supabase admin client, rejecting the
  6th request. `BurstResult` shape and fail-open contract (demo mode / DB error →
  allow) unchanged, so `app/api/simulate/route.ts` needed no edits.
- **`lib/upstash.ts`** — deleted (no longer referenced). `@upstash/*` npm deps left
  in `package.json` so reverting to Redis later is a one-file change.
- **`supabase/migrations/0002_burst_ip_index.sql`** — new index
  `simulation_usage_ip_created_idx` on `(ip_hash, created_at desc)`.

## Deviation
Plan 01 specced Upstash Redis as the rate-limit backend; founder cannot reach the
Upstash dashboard. Switched to Postgres (shared state — correct on Vercel serverless,
unlike an in-memory Map). Recorded against the Phase 1 "Rate-limit storage backend"
open decision.

## Verification (Phase 1 verify step 12 — CLOSED)
- `npx tsc --noEmit` → clean.
- 7 rapid `POST /api/simulate` from one IP: requests 1–5 returned the daily quota's
  `limit_reached` (this IP's free sim was already spent), request **6 flipped to
  `{"error":"rate_limited"}` (HTTP 429)** — only `checkBurst` produces that reason,
  confirming 5 windowed rows counted and the 6th rejected.
